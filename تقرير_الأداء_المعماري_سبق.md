## القسم الأول: ماذا يحدث عندما يدخل زائر واحد؟

### المرحلة 1: الوصول إلى الخادم
عندما يفتح الزائر `sabq.org`، يمر طلبه عبر **8 طبقات من المعالجة** قبل أن يرى أي محتوى:

| # | الطبقة | العملية | يستخدم قاعدة البيانات؟ |
|---|--------|---------|----------------------|
| 1 | CORS | التحقق من مصدر الطلب | لا |
| 2 | Helmet | إضافة رؤوس الأمان (CSP, HSTS, XSS) | لا |
| 3 | Compression | ضغط الاستجابة بـ Gzip | لا |
| 4 | APM | قياس الأداء وتسجيل الاستجابات | لا |
| 5 | Session + Passport | البحث عن جلسة المستخدم | **نعم** (sessions table) |
| 6 | Social Crawler | فحص User-Agent لروبوتات التواصل الاجتماعي | **نعم** (إذا كان روبوت) |
| 7 | SEO Injector | حقن meta tags في HTML | **نعم** (articles/categories) |
| 8 | Legacy Redirects | فحص التحويلات القديمة | **نعم** (legacy_redirects — مع كاش 60 ثانية) |

**النتيجة:** حتى الزائر العادي (غير مسجل) يمر عبر استعلام جلسة في قاعدة البيانات + فحص SEO.

### المرحلة 2: تحميل بيانات الصفحة الرئيسية
بعد تحميل HTML، المتصفح يطلب فوراً نقطتين أساسيتين:

#### 1. `GET /api/auth/user` — فحص تسجيل الدخول
- استعلام واحد لجدول `sessions` + `users`
- مخزن مؤقتاً في الذاكرة لمدة 5 دقائق

#### 2. `GET /api/homepage-lite` — المحتوى الرئيسي (الأهم)
هذا هو الطلب الأثقل. عند غياب الكاش، ينفذ **6 استعلامات متوازية**:

| # | الاستعلام | الجداول المستخدمة | عدد النتائج |
|---|-----------|-------------------|-------------|
| 1 | أخبار البطل (Hero) | articles + categories + users (3 JOINs) | 5 مقالات |
| 2 | لك (For You) | articles + categories + users | 28 مقالة |
| 3 | أخبار عاجلة | articles + categories + users | 5 مقالات |
| 4 | اختيارات المحرر | articles + categories + users | 6 مقالات |
| 5 | تحليل عميق (Deep Dive) | articles + categories + users | 6 مقالات |
| 6 | المواضيع الرائجة | articles + topics (تجميع views) | 10 مواضيع |

**المجموع:** 6 استعلامات × 3 JOINs لكل واحد = **~18 عملية ربط جداول** لكل تحميل غير مخزن.

### المرحلة 4: الاتصال المستمر (SSE)
بعد التحميل، المتصفح يفتح اتصالاً مستمراً:
- `GET /api/cache-invalidation/stream` — اتصال SSE مفتوح طوال الزيارة
- يستهلك اتصال TCP واحد من الخادم لكل زائر نشط
- يُستخدم لتحديث الأخبار العاجلة فوراً

## القسم الثاني: لماذا يحدث اختناق عند آلاف الزوار؟

### السبب 1: انفجار اتصالات قاعدة البيانات (الأخطر)
```
الحد الأقصى لاتصالات الخادم: 15 اتصال لكل Pod
عدد الـ Pods: 3 (Autoscale)
حد Neon: 80 اتصال

المعادلة:
15 × 3 = 45 اتصال متاح فقط
```

**المشكلة:** كل طلب يحتاج اتصالاً بقاعدة البيانات (للجلسات على الأقل). عند 2000+ زائر متزامن:
- 45 اتصالاً لا تكفي
- الطلبات تنتظر في الطابور (`connectionTimeoutMillis: 10s`)
- بعد 10 ثوانٍ بدون اتصال متاح = **خطأ 500**

---

### السبب 2: اتصالات SSE تستنزف الموارد
```
كل زائر = اتصال SSE مفتوح = اتصال TCP محجوز

1000 زائر متزامن = 1000 اتصال TCP مفتوح
5000 زائر متزامن = 5000 اتصال TCP مفتوح
```
**المشكلة:** الخادم له حد لعدد الاتصالات المتزامنة. اتصالات SSE لا تُغلق حتى يغادر الزائر.

---

### السبب 3: الجلسات تُخزن في قاعدة البيانات
```
كل طلب API = SELECT من جدول sessions
731,000 جلسة نشطة = جدول 516 MB

حتى الزوار غير المسجلين يحصلون على جلسة
```
**المشكلة:** جدول الجلسات يُستعلم في **كل طلب**. مع مئات الآلاف من الصفوف، يصبح بطيئاً.

---

### السبب 4: N+1 في لوحة الإدارة (إداري فقط)
```
عرض قائمة 500 مستخدم:
= 1 استعلام للقائمة
+ 500 استعلام للأدوار
+ 500 استعلام لمعلومات الموظفين
= 1,001 استعلام لصفحة واحدة!
```
**المشكلة:** هذا يستهلك اتصالات قاعدة البيانات ويبطئ بقية الطلبات.

---

### السبب 5: انتهاء صلاحية الكاش في نفس اللحظة (Thundering Herd)
```
الكاش الرئيسي: 10 دقائق
عند انتهاء الصلاحية:
- أول طلب يبدأ تحديث الكاش (6 استعلامات ثقيلة)
- الطلبات التالية تنتظر...

مع SWR: الطلبات تحصل على البيانات القديمة فوراً
لكن: طلب التحديث نفسه يحجز اتصال DB لمدة 1-2 ثانية
```
**ملاحظة:** نظام SWR الموجود يخفف المشكلة لكن لا يحلها كلياً.

WHERE title ILIKE '%كلمة البحث%'

-- المشكلة: % في البداية = لا يستخدم أي فهرس

## القسم الثالث: خريطة تدفق البيانات عند دخول زائر

```
الزائر يفتح sabq.org
        │
        ▼
   [Cloudflare CDN]
   هل الصفحة مخزنة؟
   ├── نعم → إرجاع فوري (0ms)
   └── لا ──▼
        [Middleware Chain]
        CORS → Helmet → Compression → APM
        │
        ▼
   [Session Lookup] ← DB Query #1
        │
        ▼
   [SEO Injector] ← DB Query #2 (articles/categories)
        │
        ▼
   [HTML Response] → المتصفح يبدأ التحميل
        │
        ▼
   [JavaScript Bundle] ← من Cloudflare CDN (مخزن سنة)
        │
        ▼
   [React يبدأ]
   ├── GET /api/auth/user ← DB Query #3
   ├── GET /api/homepage-lite
   │   ├── كاش موجود؟ → إرجاع فوري
   │   └── كاش منتهي؟ → 6 DB Queries (#4-#9)
   │
   └── SSE Connection ← اتصال TCP مستمر
        │
        ▼
   [الصفحة تظهر للزائر]
   زمن التحميل المعتاد: 133-172ms (مع كاش)
   زمن التحميل بدون كاش: 1500-2000ms
```

## القسم الرابع: جدول المخاطر والأولويات

| # | المشكلة | الخطورة | التأثير | الحل المقترح |
|---|---------|---------|---------|-------------|
| 1 | اتصالات DB محدودة (45) | حرجة | الموقع يتوقف عند 2000+ زائر | Redis للجلسات + زيادة Pool |
| 2 | SSE لكل زائر | عالية | استنزاف اتصالات TCP | تحويل إلى Polling كل 30 ثانية أو WebSocket مع حد أقصى |
| 3 | Session في PostgreSQL | عالية | كل طلب = استعلام DB | Redis Sessions أو JWT بدون حالة |
| 4 | N+1 في Admin Users | متوسطة | 1001 استعلام لصفحة واحدة | JOIN واحد بدل حلقات |
| 5 | بحث بدون فهرس GIN | متوسطة | بحث بطيء = حجز اتصالات | إضافة `pg_trgm` GIN index |
| 6 | SEO Injector يستعلم DB | منخفضة | استعلام إضافي لكل صفحة | كاش في الذاكرة 5 دقائق |

## القسم الخامس: التوصيات العاجلة

### أولوية 1: Redis للجلسات (تأثير فوري)
- نقل الجلسات من PostgreSQL إلى Redis
- يزيل ~50% من استعلامات قاعدة البيانات فوراً
- Redis يتحمل 100,000+ عملية/ثانية مقابل ~5,000 لـ PostgreSQL

### أولوية 2: تحديد حد SSE + Polling Fallback
- حد أقصى 500 اتصال SSE متزامن (موجود جزئياً)
- الباقي يتحول تلقائياً إلى Polling كل 30 ثانية
- يقلل اتصالات TCP بنسبة 80%+

### أولوية 3: إلغاء الجلسة للزوار غير المسجلين
- الزوار غير المسجلين لا يحتاجون جلسة في قاعدة البيانات
- إعداد `saveUninitialized: false` موجود لكن الجلسة تُنشأ عند أول تفاعل
- الحل: تأخير إنشاء الجلسة حتى تسجيل الدخول فعلياً

### أولوية 4: إصلاح N+1 في لوحة الإدارة
- استبدال 1001 استعلام بـ 3 استعلامات فقط (JOIN)
- لا يؤثر على الزوار مباشرة لكن يحرر اتصالات DB

## الخلاصة

الموقع مصمم بشكل جيد من حيث الكاش والتحميل الكسول، لكن **عنق الزجاجة الرئيسي** هو الاعتماد الكامل على PostgreSQL لكل شيء: الجلسات، الكاش، والبيانات. عند تدفق آلاف الزوار، الـ 45 اتصالاً المتاحة تنفد بسرعة.

**الحل الجذري:** فصل طبقة الجلسات والكاش عن قاعدة البيانات باستخدام Redis، مما يحرر PostgreSQL للعمل الذي صُمم له: تخزين واسترجاع البيانات فقط.
